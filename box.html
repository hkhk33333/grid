Context-Aware AI Dev Agent

Scenario
A Box engineer's deploy fails with a cryptic error in the payment service. Today, they search Slack, scan three wikis, and ping a senior engineer. With the AI Dev Agent, they ask: "Why is payment-service deploy failing with error PG_CONN_TIMEOUT?" — and get the relevant runbook, last incident resolution, and suggested fix in seconds.

What This Requires

  • Box AI platform + MCP server for content indexing (already built for customers)
  • Connectors for internal knowledge: Confluence, GitHub, Slack, PagerDuty
  • IDE plugin and CLI integration for in-flow access
  • Permissions-aware retrieval (Box's core competency)

Why Now

  • Box AI + MCP infrastructure is production-ready — apply it internally
  • AI Studio's multi-model architecture supports custom agent creation today
  • Box is "AI-first" — internal dogfooding validates the platform for customers
  • Agentic engineering platforms are the next IDP evolution (Port.io, Harness)
Strategic value: Every bug the AI Dev Agent solves internally becomes a customer-facing case study. Dogfooding de-risks the platform and generates authentic product insights competitors can't replicate.

Technical Architecture

Trigger
Engineer query via IDE / CLI / Slack
→
Retrieval
MCP server indexes internal Box content
→
AI Layer
Box AI agent with codebase context + RAG
→
Response
Actionable answer with source links

Latency

Challenge
RAG over large doc corpus can be slow. Pre-index high-value sources (runbooks, architecture docs); lazy-load Slack/comments on demand.

Cost

Challenge
LLM inference at scale is expensive. Use Box AI's existing multi-model routing: fast models for simple lookups, capable models for complex diagnosis.

Accuracy

Challenge
Stale docs lead to wrong answers. Show confidence scores and source timestamps; flag when content is >90 days old. Engineers rate responses to improve ranking.

Privacy

Challenge
Internal docs contain sensitive data. Leverage Box's permissions model — the agent only surfaces content the querying engineer has access to. No new auth layer needed.

Success Metrics

Adoption
Weekly Active Engineers
Engineers making at least one query per week. Segmented by team and tenure — high adoption among new hires signals strong onboarding value.
Quality
Resolution Rate
Percentage of queries where the engineer found their answer without escalating to another person. Measured via thumbs-up/down + follow-up tracking.
Impact
Time-to-Resolution
Time from encountering an issue to resolving it, compared against pre-agent baseline. Correlate with DORA metrics (MTTR, deploy frequency).

Why These Metrics

Internal tools succeed or fail on voluntary adoption — engineers won't use a tool that wastes their time. Weekly active engineers is the leading indicator. Resolution rate measures whether the agent actually solves problems (not just surfaces links). Time-to-resolution ties to business impact: faster unblocking means faster shipping, which the IDP team can report as platform ROI.
